# 슬라이드 기획 워크플로우 전체 설계안 (v4)

## 설계 개요

- **작성일**: 2026-03-13
- **버전**: v4
- **목적**: `/slide-planning` 워크플로우 5-Phase 전체 설계
- **핵심 변경**: 하이브리드 입력(아키텍처+세션), 단일 밀도(20-35줄, 3-5분/장)

### v3 → v4 핵심 변경사항

| 항목 | v3 (기존) | v4 (변경) | 근거 |
|------|----------|----------|------|
| **입력 소스** | 모듈 교안(`06_modules/`) 주 입력 | **아키텍처+세션 하이브리드** | 모듈은 중간 계층. 아키텍처(구조)와 세션(상세)이 더 정확한 2계층 |
| **밀도 체계** | 3단계 프리셋 (8-15/15-21/5-7줄) | **단일 밀도 20-35줄, 3-5분/장** | 사용자 지정. 고밀도 참조형에 최적화 |
| **슬라이드 수** | ~17장/세션 (50min÷3) | **~10-13장/세션** (50min÷4) | 밀도 증가 → 장당 시간 증가 → 장 수 감소 |
| **A-E 적용** | 프리셋 연동 (full/partial/bullet) | **항상 partial** (고정) | 20-35줄은 시각 증거 공간 압축 → partial이 최적 |
| **PQ3 질문** | 사용자 밀도 프리셋 선택 | **제거** (고정값) | 밀도가 고정되었으므로 질문 불필요 |
| **Layer 구조** | 3계층(통합본 상단+모듈+JSON) | **4계층(아키텍처+세션+메타데이터+JSON)** | 아키텍처 파일이 독립 Layer로 추가 |

## 1. 하이브리드 2계층 입력 전략

### 1-1. 왜 아키텍처 + 세션인가

```
┌──────────────────────────────────────────────────┐
│          01_outline/05_arch_architecture.md        │
│          (전체 지도 — 구조·흐름·정렬)               │
│  ┌─────────────────────────────────────────────┐  │
│  │ §5 매크로 구조 (Phase A-D 비율)              │  │
│  │ §5-2 일자별 개요 (Day별 주제·Phase·산출물)    │  │
│  │ §시간표 (일별×교시별 그리드)                   │  │
│  │ §7 차시 간 연결 맵 (산출물 흐름)              │  │
│  │ §8 정렬 매트릭스 (SLO↔활동↔평가)             │  │
│  └─────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
                      +
┌──────────────────────────────────────────────────┐
│   02_script/06_sessions/06_session_{NNN}.md × N   │
│   (차시별 상세 — 콘텐츠·발문·활동·평가)             │
│  ┌─────────────────────────────────────────────┐  │
│  │ 차시 헤더 (Phase, 교수 모델, SLO, 시간배분)   │  │
│  │ 도입: Hook, SLO 제시, 사전평가               │  │
│  │ 전개: 개념 설명, 코드, 활동, 발문, 형성평가     │  │
│  │ 정리: 요약, Exit Ticket, 다음 차시 연결       │  │
│  │ 발표자 노트 (타이밍, 오개념, 대안)             │  │
│  └─────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

### 1-2. 3가지 후보 비교 (결정 근거)

| 평가 기준 | 세션 파일 | 모듈 파일 | 통합본 |
|----------|---------|---------|-------|
| **콘텐츠 충분성** | ✅ 도입/전개/정리 완전 | ✅ 세션 포함 + Synthesizer | ✅ 전체 포함 |
| **처리 단위 적합** | ✅ 1세션=1슬라이드세트 | △ 4세션 묶음 | ✗ 전체 한 덩어리 |
| **컨텍스트 효율** | ✅ ~200줄/세션 | △ ~800줄/모듈 | ✗ 수천줄 |
| **품질 상태** | △ 패치 전 상태 | ✅ 패치 완료 | ✅ 최종 통합 |
| **구조 맥락** | ✗ 세션 간 관계 부재 | △ 모듈 내 관계만 | ✅ 전체 관계 |

**결론**: 세션 파일의 약점(구조 맥락 부재, 패치 전 상태)을 아키텍처 파일이 보완 → **하이브리드 최적**

### 1-3. 4계층 로드 전략

| Layer | 소스 파일 | 파싱 항목 | 용도 |
|-------|---------|----------|------|
| **L1: 구조** | `01_outline/05_arch_architecture.md` | 매크로 구조, 일자별 개요, 시간표, 연결 맵, 정렬 매트릭스 | 슬라이드 시퀀스의 뼈대 |
| **L2: 콘텐츠** | `02_script/06_sessions/06_session_{NNN}.md` × N | 차시별 도입/전개/정리, 발문, 활동, 형성평가, 발표자 노트 | 슬라이드 내용 채우기 |
| **L3: 메타데이터** | `02_script/06_write_lecture_script.md` 상단 ~100줄 | topic, target_learner, format, schedule, SLO 목록 | 코스 레벨 정보 |
| **L4: 보조 JSON** | `01/02_input_data.json` | teaching_model, tone_examples, lab_environment, keywords | 설정 보완 |

### 1-4. Fallback 체인

| 실패 시나리오 | Fallback | 비고 |
|-------------|----------|------|
| 세션 폴더(`06_sessions/`) 없음 | 모듈 폴더(`06_modules/`) 파싱 | 경고 출력 |
| 모듈 폴더도 없음 | 통합본 전체 파싱 | 경고 출력, 대용량 주의 |
| 아키텍처 파일 없음 | 구성안(`01_outline/06_write_lecture_outline.md`) 대체 | §5 로드맵 + §시간표 활용 |
| 구성안도 없음 | 오류: "`/lecture-outline` 먼저 실행" → 종료 | — |

## 2. 밀도 체계 (v4: 단일 프로파일)

### 2-1. 핵심 수치

| 항목 | 값 | 근거 |
|------|-----|------|
| **정보 밀도** | 20-35줄/슬라이드 | 사용자 지정. 핸드아웃 겸용 참조형 |
| **시간 배분** | 3-5분/슬라이드 | 고밀도 콘텐츠의 설명 시간 확보 |
| **세션당 슬라이드** | 10-13장 (50분 기준) | 50min ÷ 4min(평균) ≈ 12.5장 |
| **A-E 적용** | partial (부분) | 20-35줄은 시각 증거 공간 30-40%만 확보 가능 |
| **1 아이디어/슬라이드** | 필수 원칙 | 인지부하 이론 (Sweller), 불변 |

### 2-2. 유형별 밀도 상세

| 유형 | 줄 수 | 시간(분) | A-E | 비고 |
|------|-------|---------|-----|------|
| `title` | 3-5 | 1-2 | — | 강의명, 발표자 |
| `agenda` | 8-12 | 2-3 | — | 학습 목표 목록 |
| `section_transition` | 3-5 | 1 | — | 섹션 전환 신호 |
| `concept` | 20-30 | 3-5 | ✅ partial | assertion + 설명 텍스트 + 축소 시각자료 |
| `code` | 25-35 | 4-5 | ✅ 변형 | assertion + 코드 블록 + 상세 주석 |
| `comparison` | 20-30 | 3-5 | ✅ partial | assertion + 비교표(다축) |
| `data_insight` | 15-25 | 3-5 | ✅ partial | assertion + 차트 + 핵심 수치 |
| `image` | 10-20 | 3-4 | ✅ partial | assertion + 다이어그램 + 설명 |
| `timeline` | 15-25 | 3-4 | 변형 | 단계별 시각화 + 설명 |
| `quote` | 5-10 | 2-3 | — | 인용문 + 출처 + 맥락 |
| `summary` | 15-25 | 3-5 | ✅ 변형 | 핵심 포인트 5-7개 + 한줄 요약 |
| `activity` | 20-30 | 표시만 | 변형 | 절차 + 시간 + 자료 (활동시간 별도) |
| `quiz` | 15-25 | 3-5 | — | 문항 + 선택지 + 힌트 |

### 2-3. Assertion-Evidence Partial 적용 규칙

**partial A-E란**: 슬라이드 상단에 Assertion Headline(주장 문장)은 유지하되, Visual Evidence 비율을 60-70%에서 **30-40%**로 축소. 나머지 공간에 상세 텍스트 배치.

```
┌────────────────────────────────────┐
│ ■ Assertion Headline               │  ← 완전한 주장 문장 (1-2줄)
│   (핵심 주장 한 문장)                │
├────────────────────────────────────┤
│                                    │
│  설명 텍스트      │  시각 증거      │  ← 60:40 또는 70:30 분할
│  (상세 설명       │  (축소 다이어그램,│
│   불릿 포인트     │   핵심 차트)     │
│   코드 스니펫)    │                 │
│                   │                 │
└────────────────────────────────────┘
```

**적용 유형**: concept, comparison, data_insight, image, summary
**미적용 유형**: title, agenda, section_transition, quote, quiz
**변형 적용**: code (assertion + 코드 블록이 evidence 역할), activity (assertion + 절차)

### 2-4. 세션당 슬라이드 구성 패턴 (50분, 10-13장)

```
[도입 — 1-2장] (5-8분)
  1. 제목/동기유발 (title/quote) ─────── 2-3분
  2. 차시 학습목표 (agenda) ────────── 2-3분

[전개 — 7-9장] (30-35분)
  3. 핵심 개념 A (concept) ─────────── 3-5분
  4. 핵심 개념 A 심화 (concept/code) ── 3-5분
  5. 실습/활동 A (activity) ─────────── 표시만
  6. 중간 전환/체크 (transition/quiz) ─ 1-3분
  7. 핵심 개념 B (concept) ─────────── 3-5분
  8. 핵심 개념 B 심화 (concept/code) ── 3-5분
  9. 형성평가 (quiz) ───────────────── 3-5분

[정리 — 1-2장] (5-8분)
 10. 핵심 요약 (summary) ──────────── 3-5분
 11. 다음 차시 예고 (transition) ───── 1-2분 (선택적)
```

> **핵심 차이** (v3 대비): 전개부 슬라이드 수가 10-15장 → 7-9장으로 감소.
> 각 슬라이드의 정보 밀도가 2배 → 장당 설명 시간이 길어짐 → 장 수 자연 감소.

## 4. Phase 2: 브레인스토밍 (brainstorm-agent)

### 4-1. 개요

| 항목 | 내용 |
|------|------|
| **에이전트** | brainstorm-agent |
| **입력** | `01_input_data.json`, `01_outline/05_arch_architecture.md`, 세션 파일(샘플 2-3개) |
| **산출물** | `03_brainstorm_divergent.md`, `03_brainstorm_convergent.md`, `03_brainstorm_review.md`, `03_brainstorm_result.md` ★ |
| **초점** | 시각화 아이디어, 레이아웃 패턴, 인터랙션 요소 |

### 4-2. 전체 흐름

```
01_input_data.json + architecture + 세션 파일(샘플)
        │
        ▼
  ┌─────────────┐
  │ Step 0      │ 맥락 분석 — 콘텐츠 유형 분포, 시각화 기회 매핑
  │ Read        │ 아키텍처에서 Phase별 주제 흐름 추출
  └──────┬──────┘
         │
         ▼
  ┌─────────────┐
  │ Step 1      │ 발산적 탐색 — 시각화 아이디어 발산

  └──────┬──────┘
         │ 03_brainstorm_divergent.md
         ▼
  ┌─────────────┐
  │ Step 2      │ 수렴 — 밀도 적합성 + 시각 계층 + 인지부하 평가
  │ Read, Write │ 아이디어 우선순위화, 세션별 시각 전략 매핑
  └──────┬──────┘
         │ 03_brainstorm_convergent.md
         ▼
  ┌─────────────┐
  │ Step 3      │ 다관점 검증 — 3역할 순차 리뷰

         │ 03_brainstorm_review.md
         ▼
  ┌─────────────┐
  │ Step 4      │ 최종 통합
  │ Read, Write │ 세션별 시각화 전략 + 레이아웃 권장안
  └─────────────┘
         │ 03_brainstorm_result.md ★
```

### 4-3. Step별 상세

#### Step 0: 맥락 분석

- 아키텍처 §5 매크로 구조에서 Phase A→D 흐름 파악
- 콘텐츠 유형 분포 분석:
  - 개념 설명 비율 vs 코드 비율 vs 실습 비율
  - Phase별 이론:실습 비율 변화 (A: 70:30 → D: 10:90)
- 세션 파일 샘플(2-3개)에서 구체적 콘텐츠 패턴 파악:
  - 어떤 개념이 시각화에 적합한가
  - 코드 예시의 복잡도 수준
  - 활동의 개인/그룹 비율

#### Step 1: 발산적 탐색

6가지 발산 기법을 **슬라이드 시각화** 맥락으로 재구성:

| # | 기법 | 슬라이드 적용 |
|---|------|-------------|
| 1 | 교차영역 비유 | 추상 개념 → 일상 메타포 시각화 (예: API = 레스토랑 주문) |
| 2 | 가정 역전 | "텍스트 없이 이미지만으로 전달 가능한가?" 사고실험 |
| 3 | 규모 전환 | 한 개념의 micro(코드 한 줄) vs macro(시스템 전체) 뷰 |
| 4 | 제약 변경 | "3단어로 슬라이드 제목을 만들면?" "컬러 2개만 쓴다면?" |
| 5 | 학제간 융합 | 인포그래픽 + 기술 다이어그램 + 스토리텔링 조합 |
| 6 | SCAMPER | 기존 불릿포인트 슬라이드를 Substitute/Combine/Adapt... |

**발산 대상 (유형별)**:
- `concept` 슬라이드: 다이어그램 변형, 비유 시각화, 단계적 공개
- `code` 슬라이드: 줄별 하이라이트, before/after, 플로우차트 변환
- `comparison` 슬라이드: 2열/3열, 벤 다이어그램, 매트릭스
- `activity` 슬라이드: 단계 넘버링, QR코드, 체크리스트 시각화
- `quiz` 슬라이드: 객관식 레이아웃, 빈칸 채우기, 시각 퍼즐

#### Step 2: 수렴 및 매핑

**수렴 기준** (20-35줄 밀도에 최적화):

| 기준 | 가중치 | 설명 |
|------|--------|------|
| 밀도 적합 | 30% | 20-35줄 범위 내에서 시각 요소가 배치 가능한가 |
| 시각 계층 | 25% | Assertion → Evidence → Detail 3단계 시선 흐름 |
| 인지 부하 | 25% | 작업 기억 7±2 항목 이내, 분할 효과 적용 |
| 도구 호환 | 20% | 선택된 도구(Marp/Slidev 등)에서 구현 가능한가 |

**세션별 시각 전략 매핑**: 각 세션에 대해
- 추천 레이아웃 패턴 (2-3개 후보)
- 핵심 시각 자료 유형 (다이어그램/차트/코드블록/이미지)
- 색상/아이콘 시스템 제안

#### Step 3: 다관점 검증

| 역할 | 관점 | 핵심 질문 |
|------|------|---------|
| 시각 디자인 회의론자 | "이 시각화가 불릿포인트보다 실제로 나은가?" | 시각적 복잡도가 오히려 인지부하를 높이지 않는지 |
| 청중 대리인 | "이 슬라이드를 3-5분 안에 이해할 수 있는가?" | 20-35줄이 주어진 시간에 소화 가능한 양인지 |
| 슬라이드 정렬 중재자 | "시각화가 학습 목표 달성에 기여하는가?" | 예쁘지만 SLO와 무관한 시각 요소 제거 |

#### Step 4: 최종 통합

`03_brainstorm_result.md` 구조:
```
§1 콘텐츠 유형 분포 분석
§2 시각화 아이디어 (유형별 × 세션별)
§3 레이아웃 패턴 권장안
§4 색상/아이콘 시스템
§5 도구별 구현 가이드
§6 검증 결과 요약
```

### 5-3. Step별 상세

#### Step 0: 컨텍스트 로드 + 슬라이드 예산 산출

**슬라이드 예산 공식**:
```
기본_슬라이드_수 = session_minutes ÷ 4 (평균 4분/장)
범위 = [10, 13]  (하한 10장, 상한 13장)
세션별_조정:
  - 코드 중심 세션: 하한 쪽 (10-11장, 코드 슬라이드 장당 시간 길음)
  - 개념 중심 세션: 상한 쪽 (12-13장, 개념 슬라이드 장당 시간 짧음)
  - 실습 중심 세션: 하한 쪽 (10-11장, 활동 시간 별도)
```

**콘텐츠 유형 매트릭스**:
각 세션의 교안 내용을 분석하여 유형 분포 결정:

| 세션 유형 | concept | code | activity | quiz | 기타 |
|---------|---------|------|----------|------|------|
| 개념 중심 | 50% | 10% | 15% | 10% | 15% |
| 코드 중심 | 20% | 40% | 20% | 10% | 10% |
| 실습 중심 | 15% | 15% | 40% | 15% | 15% |
| 통합/프로젝트 | 10% | 20% | 40% | 10% | 20% |

#### Step 1: Backward Design — 프레젠테이션 레벨

**세션별 "슬라이드 스토리" 설계**:

각 세션에 대해 다음을 정의:
1. **인지 여정**: 이 세션을 마친 후 학습자가 알아야 할 것 (SLO 기반)
2. **시각 내러티브**: 슬라이드 흐름이 만드는 이야기 (도입→전개→정리)
3. **핵심 전환점**: 언제 새로운 개념으로 넘어가는가

**SLO → 슬라이드 유형 매핑**:

| Bloom's 수준 | 주로 매핑되는 슬라이드 유형 |
|-------------|----------------------|
| 기억 | `concept` (정의/용어) |
| 이해 | `concept`, `comparison`, `image` |
| 적용 | `code`, `activity` |
| 분석 | `comparison`, `data_insight` |
| 평가 | `quiz`, `activity` |
| 창조 | `activity` (프로젝트) |

#### Step 2: 슬라이드 시퀀스 설계

각 세션에 대해 슬라이드 목록 생성:

```
세션 {N}: {차시 제목}
├─ 슬라이드 수: {10-13}장
├─ 시간 배분: 도입 {5-8}분 / 전개 {30-35}분 / 정리 {5-8}분
└─ 슬라이드 목록:
   #  유형              제목(가제)          시간  SLO
   1  title             {차시 제목}         2분   —
   2  agenda            {학습 목표}         3분   전체
   3  concept           {개념 A 주장}       4분   SLO-1
   4  code              {코드 예시}         5분   SLO-1
   5  activity          {실습 A}            표시  SLO-1
   6  section_transition {전환}             1분   —
   7  concept           {개념 B 주장}       4분   SLO-2
   8  comparison        {A vs B 비교}       4분   SLO-2
   9  quiz              {형성평가}          4분   SLO-1,2
  10  summary           {핵심 요약}         4분   전체
  11  section_transition {다음 차시 예고}    2분   —
```

#### Step 3: 4중 검증

| # | 검증 항목 | 기준 | FAIL 조건 |
|---|---------|------|----------|
| 1 | 슬라이드 수 | 세션당 10-13장 | 범위 이탈 |
| 2 | 시간 적합 | Σ(슬라이드 시간) ≤ session_minutes | 총 시간 초과 |
| 3 | 밀도 호환 | 모든 슬라이드 20-35줄 가능 | concept/code 유형에서 35줄 초과 예상 |
| 4 | SLO 커버리지 | 모든 SLO가 ≥1개 슬라이드에 매핑 | 매핑 누락 |

### 5-4. 산출물 구조 (05_arch_slide_structure.md)

```

  │ Step 2      │ 5영역 가중치 심층 검토
  │ Read        │ 25개 체크 항목, 0~10점 채점
  └──────┬──────┘

  └─────────────┘
```

### 7-3. 5영역 가중치 체크리스트 (v4 업데이트)

| 영역 | 가중치 | 체크 항목 (5개) |
|------|--------|---------------|
| **정보 밀도** | 25% | ①줄 수 준수(20-35줄) ②1아이디어/슬라이드 ③인지부하 적정 ④텍스트:시각 비율 ⑤코드 슬라이드 밀도 |
| **시각 계층** | 20% | ①A-E partial 적용 ②시선 흐름(Assertion→Evidence→Detail) ③레이아웃 일관성 ④색상/아이콘 체계 ⑤전환 슬라이드 배치 |
| **학습목표 정렬** | 25% | ①SLO 완전 커버리지 ②Bloom's↔유형 정합 ③평가 슬라이드 배치 ④활동 슬라이드↔교안 정합 ⑤핵심/중요/보충 반영 |
| **슬라이드 수** | 15% | ①세션당 10-13장 ②시간 적합(3-5분/장) ③도입:전개:정리 비율 ④전환 슬라이드 과다/부족 ⑤전체 슬라이드 수 적정 |
| **가독성** | 15% | ①폰트 크기 가이드 ②대비/접근성 ③발표자 노트 충실도 ④도구 호환성 ⑤마크다운 형식 정합 |

### 7-4. 판정 체계

| 판정 | 점수 범위 | 후속 조치 |
|------|---------|---------|
| **APPROVED** | 8.0+ | 완료. `/slide-generation` 진행 가능 |
| **APPROVED WITH NOTES** | 6.0-7.9 | P3 항목 표시, 사용자 선택에 따라 경량 수정 |
| **REVISE** | 4.0-5.9 | writer-agent 수정 모드 → 재검토 |
| **REVISE MAJOR** | 0-3.9 | architecture-agent부터 재실행 → 재검토 |

**자동 REVISE 조건**:
- 영역 1(정보 밀도) 또는 3(학습목표 정렬) 평균 5.0 미만
- QM 기준 4·5 미충족

### 7-5. REVISE 루프 로직

```
revision_count = 0

[Phase 5 완료]
  │
  ├─ APPROVED (8.0+) → 종료
  │
  ├─ APPROVED WITH NOTES (6.0-7.9)
  │     → 사용자에게 P3 항목 표시
  │     → "반영하겠습니까?" → 예: writer-agent 경량 수정 → 종료
  │
  ├─ REVISE / REVISE MAJOR
  │     → IF revision_count >= 1: STOP & CONSULT
  │     → revision_count += 1
  │     → restart_phase 파싱:
  │       Phase 4 → writer-agent 수정 모드
  │       Phase 3 → architecture-agent 수정 → writer-agent 재작성
  │     → Phase 5 재실행
```
